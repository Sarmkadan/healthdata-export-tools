# MetricCorrelationDto

Represents a computed correlation between two health metrics over a defined analysis window. This type encapsulates the statistical relationship (coefficient, strength, direction), contextual metadata such as sample count and interpretation, and any derived cross-metric insights. It is designed as a self-contained data transfer object for serialization and API responses within the healthdata-export-tools pipeline.

## API

### `Pair`
`public required CorrelationPair Pair`

Identifies the two metrics being correlated. The `CorrelationPair` value object holds the metric keys and their display labels, establishing the subject of the analysis.

### `Coefficient`
`public required double Coefficient`

The computed correlation coefficient (e.g., Pearson’s *r* or Spearman’s *ρ*). Values range from -1.0 (perfect negative correlation) to 1.0 (perfect positive correlation), with 0.0 indicating no linear relationship.

### `Strength`
`public required CorrelationStrength Strength`

Categorical classification of the absolute coefficient magnitude. Typical values include `Weak`, `Moderate`, `Strong`, and `VeryStrong`, derived from predefined thresholds applied to `Coefficient`.

### `Direction`
`public required CorrelationDirection Direction`

Indicates whether the relationship is `Positive` (both metrics move in the same direction), `Negative` (they move in opposite directions), or `None` (no directional trend).

### `SampleCount`
`public required int SampleCount`

The number of paired data points used in the correlation calculation. A higher count increases statistical reliability; low counts may produce spurious results.

### `Interpretation`
`public required string Interpretation`

A human-readable summary of the correlation result, combining strength, direction, and clinical or analytical context. This string is generated by the analysis engine and is suitable for direct display in reports.

### `AnalysisPeriodStart`
`public required DateOnly AnalysisPeriodStart`

The inclusive start date of the time window over which the correlation was computed.

### `AnalysisPeriodEnd`
`public required DateOnly AnalysisPeriodEnd`

The inclusive end date of the time window over which the correlation was computed.

### `IsSignificant`
`public bool IsSignificant`

Indicates whether the correlation passes the configured statistical significance threshold (e.g., *p*-value < 0.05). When `false`, the observed relationship may be due to random chance and should be interpreted with caution.

### `Title`
`public required string Title`

A concise, display-ready title for this correlation result, typically combining the metric names and the primary finding (e.g., “Steps vs. Resting Heart Rate: Moderate Negative Correlation”).

### `Description`
`public required string Description`

A longer narrative description providing context, potential implications, and limitations of the correlation. Suitable for tooltips or detail views.

### `RelatedPair`
`public required CorrelationPair RelatedPair`

A secondary or alternative metric pair that exhibits a related correlation pattern. This enables surfacing of companion findings without requiring a separate query.

### `Severity`
`public required InsightSeverity Severity`

The importance or urgency level assigned to any insight derived from this correlation. Values follow the `InsightSeverity` enumeration (e.g., `Info`, `Warning`, `Critical`) and drive UI highlighting and notification behavior.

### `Recommendation`
`public required string Recommendation`

Actionable guidance based on the correlation result. May suggest further investigation, lifestyle adjustments, or data collection improvements.

### `AnalysisId`
`public required string AnalysisId`

A unique identifier for the analysis run that produced this correlation. Used for traceability, caching, and deduplication across exports.

### `GeneratedAt`
`public required DateTimeOffset GeneratedAt`

The timestamp (with UTC offset) when this correlation result was computed. Enables consumers to determine data freshness and ordering.

### `WindowDays`
`public required int WindowDays`

The length of the analysis window in calendar days. Together with `AnalysisPeriodStart` and `AnalysisPeriodEnd`, this defines the temporal scope of the correlation.

### `Correlations`
`public required IReadOnlyList<MetricCorrelationDto> Correlations`

A collection of related correlation results, enabling hierarchical or grouped representations. May contain sub-correlations for segmented time windows or filtered subsets of the same metric pair.

### `Insights`
`public required IReadOnlyList<CrossMetricInsightDto> Insights`

Cross-metric insights derived from this correlation and potentially other related analyses. Each `CrossMetricInsightDto` encapsulates a specific finding, its severity, and supporting evidence.

## Usage

### Example 1: Serializing a correlation result for an API response

```csharp
var correlation = new MetricCorrelationDto
{
    Pair = new CorrelationPair("steps", "resting_heart_rate", "Steps", "Resting Heart Rate"),
    Coefficient = -0.62,
    Strength = CorrelationStrength.Moderate,
    Direction = CorrelationDirection.Negative,
    SampleCount = 90,
    Interpretation = "A moderate negative correlation suggests that higher daily step counts are associated with lower resting heart rate.",
    AnalysisPeriodStart = new DateOnly(2025, 1, 1),
    AnalysisPeriodEnd = new DateOnly(2025, 3, 31),
    IsSignificant = true,
    Title = "Steps vs. Resting Heart Rate",
    Description = "Over a 90-day period, daily steps and resting heart rate showed a moderate inverse relationship.",
    RelatedPair = new CorrelationPair("steps", "sleep_duration", "Steps", "Sleep Duration"),
    Severity = InsightSeverity.Info,
    Recommendation = "Maintain or increase daily step count to support cardiovascular efficiency.",
    AnalysisId = "corr-20250401-001",
    GeneratedAt = DateTimeOffset.UtcNow,
    WindowDays = 90,
    Correlations = Array.Empty<MetricCorrelationDto>().AsReadOnly(),
    Insights = Array.Empty<CrossMetricInsightDto>().AsReadOnly()
};

string json = JsonSerializer.Serialize(correlation, new JsonSerializerOptions { WriteIndented = true });
Console.WriteLine(json);
```

### Example 2: Filtering significant correlations from a batch result

```csharp
IReadOnlyList<MetricCorrelationDto> allCorrelations = GetCorrelationBatch(analysisId: "corr-20250401-001");

var significant = allCorrelations
    .Where(c => c.IsSignificant && c.SampleCount >= 30)
    .OrderByDescending(c => Math.Abs(c.Coefficient))
    .ToList();

foreach (var corr in significant)
{
    Console.WriteLine($"[{corr.Severity}] {corr.Title}: {corr.Interpretation}");
    Console.WriteLine($"  Recommendation: {corr.Recommendation}");
    Console.WriteLine($"  Period: {corr.AnalysisPeriodStart} – {corr.AnalysisPeriodEnd} ({corr.WindowDays} days, {corr.SampleCount} samples)");
}
```

## Notes

- **Required members**: All `required` properties must be initialized during construction. Failure to set any of them will result in a compile-time error when using object initializer syntax or a run-time exception if constructed via reflection or deserialization without supplying values.
- **`Correlations` and `Insights` immutability**: Both collections are exposed as `IReadOnlyList<T>`. Consumers cannot modify the lists after construction. The underlying implementation should be a read-only wrapper around an array or an immutable list to prevent accidental mutation.
- **Statistical validity**: `IsSignificant` is a boolean flag set by the analysis pipeline. When `false`, `Strength`, `Direction`, and `Interpretation` may still contain computed values, but they should not be treated as reliable. Always check `IsSignificant` and `SampleCount` before surfacing findings to end users.
- **Thread safety**: This type is a plain data transfer object with no internal synchronization. Instances are safe for concurrent reads after full construction. Concurrent writes or partial mutations during construction are not safe and must be guarded externally.
- **`RelatedPair` vs. `Pair`**: `RelatedPair` is a secondary correlation pair that may share one metric with `Pair` or represent an entirely different relationship discovered during the same analysis run. It is informational and does not replace `Pair` as the primary subject.
- **`WindowDays` consistency**: `WindowDays` should equal the inclusive difference between `AnalysisPeriodEnd` and `AnalysisPeriodStart` plus one day. Discrepancies may occur if the analysis engine uses non-contiguous data; consumers should rely on the date range for display and use `WindowDays` as a nominal duration.
- **Deserialization**: When deserializing from JSON or other formats, ensure the serializer supports `required` semantics (e.g., `System.Text.Json` with `JsonRequired` configuration or a custom contract resolver). Missing required properties will cause deserialization to fail.
